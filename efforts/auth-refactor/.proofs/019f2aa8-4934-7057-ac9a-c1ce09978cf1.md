# Proof: add-guest flow (eligible tertiary listing, PIN requirement, confirm gating)

## Criterion

The add-guest flow lists eligible tertiary users, requires PIN entry for each
selected guest that needs one, and prevents confirmation until required PINs
are valid.

## How proved

Sandbox stack, `PROOF_FLOW=group-pin` (`e2e/flows/group-pin.mjs`), visitor
account (Carol is a tertiary/guest with pin `5678`), under
`PROOF_RUN_ID=authproof-20260704T202336`.

## Evidence

- Flow step 3 opens the "+ Add guest" modal (`.guest-modal`) and asserts Carol
  is listed there (`guestModal.locator('button.guest-card', { hasText: 'Carol'
  })`), confirming eligible tertiary users are listed. The add-guest modal
  itself is selection-only (no PIN field there by design — PIN is collected
  at Continue time), and the flow explicitly asserts no `input[type=password]`
  exists in that modal.
- `artifacts/authproof-20260704T202336/2026-07-05T00-26-12-014Z-group-pin/group-pin-prompt.png`
  — after clicking "Continue" with Carol selected, the continue-time "Enter
  guest PINs" modal opens showing Carol's avatar and a "Carol's PIN" password
  input, with "Confirm PINs" present. The flow asserts this confirm button is
  DISABLED until a PIN is typed (`confirmPinsBtn.isDisabled()` check before
  screenshotting), i.e. this screenshot itself is the disabled-until-input
  state.
- `artifacts/authproof-20260704T202336/2026-07-05T00-26-12-014Z-group-pin/group-pin-wrong.png`
  — an incorrect PIN ("0000") is submitted; the server's
  `POST /api/group/verify-pins` returns 403 and the still-open modal shows the
  error "Incorrect or missing pin for Carol." beneath the PIN field — the
  group is NOT allowed to form and the modal does not close, confirming
  confirmation is prevented until the PIN is valid.
- `artifacts/authproof-20260704T202336/2026-07-05T00-26-12-014Z-group-pin/group-pin-success.png`
  — retyping the correct PIN (`5678`) in the same modal succeeds, the modal
  closes, and the main app renders — confirming a valid PIN unblocks
  confirmation and group formation.

## Verdict

Proved. Screenshots and flow assertions together show eligible tertiary
listing, PIN requirement at continue-time, and confirm-button gating on PIN
validity (both the disabled-until-typed state and the reject-until-correct
state are visually confirmed).
