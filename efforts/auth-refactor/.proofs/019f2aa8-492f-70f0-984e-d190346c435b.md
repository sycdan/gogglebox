# Proof: config schema (accounts / access_tokens) + auto-migration

## Criterion

Config supports `accounts[account_key] = account_config` and
`access_tokens[token] = account_key`, validates token uniqueness, and migrates
older supported config versions automatically on startup.

## How proved

Not a UI criterion — proved by reading the passing unit-test suite
(`docker compose run --rm test`, confirmed clean in
`efforts/auth-refactor/.outputs/019f2f9e-2acd-735e-bbce-7af071a5a490.md`:
145 tests, 141 pass, 0 fail, 4 skipped) plus the test source itself.

### `accounts[account_key]` / `access_tokens[token] -> account_key` shape

- `src/server/config.test.ts:46` — `'buildEffectiveConfig reads a valid
  schemaVersion-2 config and stamps provenance'` builds a v2 config with
  `accounts: { house1, house2 }` and `access_tokens: { 'token-1': 'house1',
  'token-2': 'house2' }` and asserts
  `effective.accessTokens` deep-equals that exact token->account-key map
  (`config.test.ts:58`) and `Object.keys(effective.accounts)` contains both
  keys (`config.test.ts:57`).

### Token uniqueness validation

- `src/server/configMigrations.test.ts:166` — `'migrate1to2 de-duplicates a
  repeated password into a unique token (warns)'`: two v1 accounts sharing the
  same password both migrate to v2 `access_tokens`, and the migration renames
  the second occurrence to a unique token (`same-pw-second`) rather than
  silently colliding, asserting `result.access_tokens` deep-equals
  `{ 'same-pw': 'first', 'same-pw-second': 'second' }` and that a warning
  mentioning `same-pw-second` is emitted.
- `src/server/configMigrations.test.ts:186` — `'migrate1to2 rechecks
  duplicate-password fallback tokens before assigning them'`: a harder case
  where the de-duplicated fallback token itself collides with a third
  account's real password; the migration re-checks the fallback and renames
  again (`same-pw-third-2`), proving uniqueness is enforced transitively, not
  just pairwise.
- `src/server/config.test.ts:254` — `'validateAndResolveConfig drops an
  access token pointing at a missing account (warn)'` and `:267` — `'...warns
  about an account with no access token (unreachable)'` cover the referential
  side of the `access_tokens[token] -> account_key` mapping (orphaned token,
  and an account unreachable for want of one).
- `src/server/config.test.ts:281` — `'validateAndResolveConfig fails fast only
  when the result is unusable'` asserts `validateAndResolveConfig` throws
  `/no access tokens remain/` when every token is stripped away, so a config
  that resolves to zero usable tokens cannot silently start.

### Migration chain (auto-migrates older config versions on startup)

- `src/server/configMigrations.test.ts:28` — `'detectSchemaVersion: missing or
  invalid => 0'`.
- `src/server/configMigrations.test.ts:35`, `:51`, `:69`, `:77`, `:86`, `:95`
  — `migrate0to1` tests: maps legacy member UUIDs to Jellyfin names, synthesizes
  an account from household/`PORTAL_*` creds, synthesizes a default account
  (with a warning) when there are no creds at all, warns+skips unmapped UUIDs,
  and drops the obsolete `groups[]` presets (with a warning) while carrying
  playback/recommendations forward.
- `src/server/configMigrations.test.ts:134` — `'migrate1to2 maps account_key =
  username and access token = password'`, `:142` — tier mapping
  (`pin_required` visible users -> tertiary, others -> secondary, primaries
  empty), `:152` — `'writes ALL THREE tier lists explicitly so wildcards
  cannot widen v1 visibility'`.
- `src/server/configMigrations.test.ts:235` — `'runMigrationChain walks a
  0-source config through 0 -> 1 -> 2'`: an end-to-end legacy (no
  `schemaVersion`) config walks the full chain and lands on
  `CURRENT_SCHEMA_VERSION` with the expected `accounts`/`access_tokens` shape.
- `src/server/configMigrations.test.ts:251` — `'runMigrationChain leaves an
  already-current config unchanged'` — a current-schema config is a no-op
  through the chain (idempotent).
- `src/server/configMigrations.test.ts:266` — `'runMigrationChain fails fast
  when the source schema is newer than the image'` — the one case the chain
  refuses to auto-migrate is a schema **newer** than the running image
  supports (by design; this is a fail-fast, not a manual-migration
  requirement for any *older* config).
- `src/server/config.test.ts:87` — `'buildEffectiveConfig auto-migrates a
  schemaVersion-1 config forward to v2'` and `:123` — `'...auto-migrates a
  legacy (no schemaVersion) config through 0 -> 1 -> 2'` exercise the same
  chain through the public `buildEffectiveConfig` entry point used at server
  startup, confirming no manual migration step is ever required — the running
  image always migrates forward automatically.
