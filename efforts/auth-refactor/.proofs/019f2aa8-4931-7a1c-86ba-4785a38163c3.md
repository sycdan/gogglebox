# Proof: portal access-token login (accept/reject/persist)

## Criterion

Portal login accepts a valid access token, rejects invalid tokens with a clear
error, and stores the token locally until logout clears it.

## How proved (sandbox)

Sandbox stack (`./scripts/sbx.sh`), `PROOF_FLOW=group-pin`
(`e2e/flows/group-pin.mjs`), run under `PROOF_RUN_ID=authproof-redo-20260705`.
Fresh sandbox provisioned in this session's own worktree/volumes (prior run's
volumes/screenshots did not survive worktree teardown).

The flow logs out the auto-logged-in household session, defeats client
auto-login by patching `GET /api/session` (`portalAutoLoginEnabled: false`),
reloads to the logged-out state, and explicitly fills the single access-token
field with the visitor account's token (`sbx-visitor-token`), gating success on
an observed `POST /api/auth/login` response of `200` followed by the "Pick the
group" picker rendering.

## Evidence (copied into this effort's `.proofs/`, not `./artifacts`)

- `efforts/auth-refactor/.proofs/019f2fc4-08ab-7693-8430-36acfd468215.png`
  (orig. `group-pin-login.png`) — the logged-out portal shows the single
  "Access token" field and "Enter gogglebox" button (accept path UI).
- Flow log: `explicit visitor token login -> 200; waiting for picker` then
  `plain viewer cards = ["Dave"]` — confirms the valid token was accepted and
  the account-scoped picker rendered (visitor account: Dave secondary, Carol
  guest, no primaries).
- Persistence: the harness's own `startSession` (`e2e/lib/session.mjs`) fills
  the token, submits, and asserts the login form detaches and a real
  authenticated element ("Log out" button) appears — i.e. the token is
  retained in the running session (no forced re-login) for the remainder of
  the flow, including across the picker, guest-add, and PIN-modal steps in
  this same run. The group-pin flow's own logout (`Log out` button click at
  the top of the flow) is what clears the previously stored household token
  and forces the login form to reappear — the screenshot above shows this
  logged-out state, demonstrating logout clears the stored token.

## Gap — invalid-token rejection (still open)

No screenshot exists of the invalid-token branch. `e2e/flows/group-pin.mjs`
(re-inspected in this pass; unchanged since the prior session) only ever
submits the valid `sbx-visitor-token`, and no other flow module under
`e2e/flows/` submits a deliberately wrong access token either. Rejection of an
*invalid* access token is therefore still NOT exercised or screenshotted by
any current flow. Per this prover's constraints (read-only on `src/`/`e2e/`),
this cannot be closed without `gogglebox-builder` adding a flow step that:

1. submits a bogus/garbage access token at the login form,
2. asserts the `POST /api/auth/login` response is rejected (4xx),
3. asserts a `.error` message renders and the login form remains (no picker),
4. screenshots that state (e.g. `shoot(page, 'group-pin-invalid-token')`).

## Verdict

Partially proved (unchanged from the prior pass on substance, now with
durable copied evidence). Valid-token accept, picker routing per account, and
logout-then-relogin (implying local persistence + logout-clears) are
confirmed visually with evidence that survives worktree teardown.
Invalid-token rejection is still NOT visually confirmed — needs an added flow
step from `gogglebox-builder` before this criterion can be marked fully
proved.
