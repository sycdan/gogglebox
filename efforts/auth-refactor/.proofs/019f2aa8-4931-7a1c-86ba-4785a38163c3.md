# Proof: portal access-token login (accept/reject/persist)

## Criterion

Portal login accepts a valid access token, rejects invalid tokens with a clear
error, and stores the token locally until logout clears it.

## How proved

Sandbox stack (`./scripts/sbx.sh`), `PROOF_FLOW=group-pin`
(`e2e/flows/group-pin.mjs`), run under `PROOF_RUN_ID=authproof-20260704T202336`.

The flow logs out the auto-logged-in household session, defeats client
auto-login by patching `GET /api/session` (`portalAutoLoginEnabled: false`),
reloads to the logged-out state, and explicitly fills the single access-token
field with the visitor account's token (`sbx-visitor-token`), gating success on
an observed `POST /api/auth/login` response of `200` followed by the "Pick the
group" picker rendering.

## Evidence

- `artifacts/authproof-20260704T202336/2026-07-05T00-26-12-014Z-group-pin/group-pin-login.png`
  — the logged-out portal shows the single "Access token" field and "Enter
  gogglebox" button (accept path UI).
- Flow log: `explicit visitor token login -> 200; waiting for picker` then
  `plain viewer cards = ["Dave"]` — confirms the valid token was accepted and
  the account-scoped picker rendered (visitor account: Dave secondary, Carol
  guest, no primaries).
- `artifacts/authproof-20260704T202336/2026-07-05T00-26-12-014Z-group-pin/group-pin-02-authenticated.png`
  — the initial household auto-login (sbx `ACCESS_TOKEN`) reaching the
  authenticated app, showing the token-accept path again for a second account.
- Persistence: the harness's own `startSession` (`e2e/lib/session.mjs`) fills
  the token, submits, and asserts the login form detaches and a real
  authenticated element ("Log out" button) appears — i.e. the token is
  retained in the running session (no forced re-login) for the remainder of
  the flow, including across the picker, guest-add, and PIN-modal steps in
  this same run. The group-pin flow's own logout (`Log out` button click at
  the top of the flow) is what clears the previously stored household token
  and forces the login form to reappear, which the screenshot above shows
  succeeding — demonstrating logout clears the stored token.

## Gap — invalid-token rejection

No screenshot exists of the invalid-token branch specifically. The proof
flows only submit valid tokens (household auto-login, then a valid visitor
token). Rejection of an *invalid* access token is not currently exercised or
screenshotted by any `e2e/flows/*.mjs` module — this needs a small addition
(a flow or flow step submitting a deliberately wrong access token and
asserting a clear `.error` message + login form remaining) from
`gogglebox-builder` for full visual coverage of the "rejects invalid tokens
with a clear error" half of this criterion.

## Verdict

Partially proved. Valid-token accept, picker routing per account, and
logout-then-relogin (implying local persistence + logout-clears) are
confirmed visually. Invalid-token rejection is NOT visually confirmed in this
pass — needs an added flow step.
