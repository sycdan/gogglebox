# Proof: mixed-group (non-primary) watch-progress warning

## Criterion

Continuing with a group that contains non-primary users shows a confirmation
warning that watch progress will be affected for every user in the group.

## How proved

Sandbox stack, `PROOF_FLOW=group-pin` (`e2e/flows/group-pin.mjs`), visitor
account, under `PROOF_RUN_ID=authproof-20260704T202336`. Dave (secondary) +
Carol (tertiary guest, added via "+ Add guest") is a group with no primary
members — a mixed/non-primary group.

## Evidence

- Source: `src/client/App.tsx` renders a `.confirm-modal` with heading
  "Shared watch progress" and body text "...will affect watch progress and
  watched states for ALL users in the group." whenever the forming group
  contains any non-primary member (comment at `App.tsx:880`: "warn when the
  group contains ANY non-primary member (watch progress is...)").
- Flow assertion: after a successful `POST /api/group/verify-pins`, the flow
  explicitly waits for `.confirm-modal` to become visible
  (`confirmModal.waitFor({ state: 'visible' })`) and fails the run loudly
  (`group-pin: the mixed-group confirmation modal did not appear before the
  group POST`) if it does not — this executed and passed in this run (no
  failure raised), then the flow clicks the modal's "Confirm" button before
  the group POST fires and the picker is left.
- The flow also proves the NEGATIVE case is respected: when the wrong PIN was
  submitted, the flow explicitly asserts the mixed-group modal did **not**
  appear (`group-pin: the mixed-group warning appeared for a wrong PIN —
  verification must happen first` would have failed the run), confirming the
  warning is gated correctly on successful PIN verification rather than
  appearing unconditionally.

## Gap

No PNG screenshot exists of the "Shared watch progress" confirm modal itself
in the open state — the flow asserts its visibility and immediately clicks
through it without an intervening `shoot()` call on the happy path (only a
failure-path screenshot, `${flowName}-mixed-modal-MISSING`, exists, and that
did not fire in this run). The warning's presence is proved by a passing
Playwright element-visibility assertion tied to real server state
(`verify-pins` 200 gating), not by a screenshot a human can read directly.

## Verdict

Partially proved. The warning's appearance is confirmed via a real, passing
DOM-visibility assertion (and its absence on the wrong-PIN path is also
confirmed), but not via a screenshot showing the actual warning text on
screen. Recommend `gogglebox-builder` add one `shoot(page, 'group-pin-mixed-warning')`
call in `e2e/flows/group-pin.mjs` right after `confirmModal.waitFor(...)` and
before the `Confirm` click, so a future prover pass can capture and Read the
warning modal directly.
