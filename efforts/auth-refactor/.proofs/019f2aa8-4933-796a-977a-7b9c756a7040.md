# Proof: account user rendering (primary preselected, secondary visible, add-guest sizing)

## Criterion

Account users render as primary users selected by default with avatars,
secondary users visible and unselected by default, and an add-guest control
matching avatar sizing.

## How proved

Sandbox stack, two flows under one `PROOF_RUN_ID=authproof-20260704T202336`:

- `PROOF_FLOW=group-alias` (`e2e/flows/group-alias.mjs`) against the
  `household` account (Alice + Bob primaries, Carol + Dave secondaries, no
  guests).
- `PROOF_FLOW=group-pin` (`e2e/flows/group-pin.mjs`) against the `visitor`
  account (Dave secondary, Carol tertiary/guest, no primaries).

## Evidence

- `artifacts/authproof-20260704T202336/2026-07-05T00-26-31-139Z-group-alias/group-alias-picker.png`
  — household picker: Alice and Bob render with round lettered avatars ("A",
  "B") and a highlighted/selected border (primary, preselected by default);
  Carol and Dave render as visible but visually unselected (no highlighted
  border) round avatar cards ("C", "D") — matching "secondary visible and
  unselected by default".
- `artifacts/authproof-20260704T202336/2026-07-05T00-26-12-014Z-group-pin/group-pin-picker.png`
  — visitor picker: only Dave (secondary) renders as a plain avatar card,
  unselected; a "+" round "Add guest" control renders alongside it at the
  same circular size/diameter as the Dave avatar, confirming the add-guest
  control matches avatar sizing. Carol (the visitor's tertiary/guest) does
  NOT render as a plain card, consistent with tertiary users only being
  reachable via "+ Add guest".

Flow-log corroboration: `group-alias: visible viewer cards = ["Alice","Bob"]`
region asserted present, and `group-pin: plain viewer cards = ["Dave"]` with
an explicit assertion that the "+ Add guest" card element
(`button.add-guest-card`) exists.

## Verdict

Proved. Both screenshots visually confirm primary preselection with avatars,
secondary visible/unselected rendering, and add-guest control sizing parity
with avatar circles.
