# Proof: group selection + Jellyfin player handoff for token-authenticated accounts

## Criterion

Existing group selection and Jellyfin player handoff continue to work for
token-authenticated accounts in both sandbox proof and real-data UAT where
available.

## How proved (sandbox)

Sandbox stack, three flows under `PROOF_RUN_ID=authproof-redo-20260705` (fresh
sandbox re-provisioned in this session's own worktree/volumes; the prior
session's `./artifacts` evidence did not survive worktree teardown):

- `PROOF_FLOW=group-alias` (`e2e/flows/group-alias.mjs`) — household account
  (token-authenticated via sbx `ACCESS_TOKEN` auto-login, itself an access
  token) creates a group (Alice + Bob), the alias "Alice + Bob" is saved and
  reused without duplicating the group.
- `PROOF_FLOW=player-handoff` (`e2e/flows/player-handoff.mjs`), run against
  the same-origin proxy (`-e PROOF_URL=http://proxy:8080`) as required for
  origin-matched localStorage seeding. (The proxy's host port 8080 was
  occupied by an unrelated, already-running `gogglebox-proxy-1` stack on this
  machine; the proxy container in this session's own compose project was
  brought up with its host port publish cleared via a local, non-committed
  `docker-compose.local.yml` override so the two stacks would not conflict —
  this does not change the app's own configuration, only how the proxy
  container's port is exposed to the host for this proof run.)

## How proved (UAT — real Jellyfin)

Ran against the real Jellyfin at `.env.uat` / `config.uat.json`
(`JELLYFIN_URL=http://htpc.lan:8096`, account `wonderland` / token
`white-rabbit`, `Alice` primary + `Bob` tertiary) via
`docker compose -f docker-compose.yml -f docker-compose.uat.yml` (the
`./scripts/uat.sh` wrapper, run explicitly with `--no-deps` for the `proof`
container to avoid re-triggering an unrelated proxy container elsewhere on
this host — see note above; `server`/`client` were still brought up fresh via
`./scripts/uat.sh up -d` / `up -d --force-recreate client` first). The real
Jellyfin at `http://htpc.lan:8096` was confirmed reachable
(`GET /System/Info/Public` -> `HTTP 200`) and the server logs show
`[startup] derived effective config (schemaVersion 2, ...)` — i.e. it
authenticated and built config from the real instance, not a stub.

- `PROOF_FLOW=group-alias` — FAILED at the assertion step
  (`group-alias: picker must show Alice and Bob (saw [Alice])`), because that
  flow hard-codes the sandbox fixture's two-primary household
  (`Alice + Bob`); the real UAT config only has one primary (`Alice`; `Bob` is
  a tertiary/pin user), so the flow's own fixture assumption does not hold
  against real data — this is a flow/fixture mismatch, not an app defect. The
  screenshot captured before the failure
  (`efforts/auth-refactor/.proofs/019f2fc8-094f-78ec-a15b-e1623bb0f2e5.png`,
  orig. `group-alias-02-authenticated.png`) itself is still useful evidence:
  it shows token-authenticated auto-login reaching the real "Pick the group"
  picker with Alice rendered as a real primary (real avatar art, not a
  sandbox stub) and "+ Add guest" present for the tertiary Bob.
- `PROOF_FLOW=player-focus` (`e2e/flows/player-focus.mjs`) — selected the
  first (and only) viewer, Alice, and continued into the main app
  successfully (group selection worked end-to-end against real data), but
  then FAILED with a Playwright **strict-mode violation** (not an auth or
  app-behavior failure): `locator('.media-card button').filter({ hasText:
  /^Play$/ }).first().or(...)` resolved to 2 elements (a `Resume` button and a
  `Play` button both matched across different cards in the real, larger
  library), so the flow's single generic locator couldn't disambiguate which
  card to click for the player-handoff step. The screenshot captured at that
  point
  (`efforts/auth-refactor/.proofs/019f2fc8-0950-7991-a38c-31860423f1c9.png`,
  orig. `player-focus-03-no-play-button.png`) shows the main app rendering
  fully against the real Jellyfin — real posters/titles ("(500) Days of
  Summer", "Tree Trunks", "In a Lonely Place"), a real "Resume" card at 40%
  progress, and working "Play"/"Resume"/"Ignore" controls — i.e. group
  selection and library rendering are proven working against real data, but
  the flow's player-launch click-through itself could not be exercised
  end-to-end due to this selector ambiguity against real (non-fixture)
  content.

## Verdict

Proved for the sandbox half (group selection + full player handoff, including
actual playback progress in the `/player` iframe, all working for
token-authenticated accounts).

Partially proved for the UAT/real-data half: real-Jellyfin reachability,
token-authenticated login, and group selection into a fully-rendered main app
are all confirmed with screenshots against the developer's actual Jellyfin.
The player-launch **click-through** into `/player` was NOT completed against
real data in this pass — both available generic flows either assume
sandbox-only fixture shapes (`group-alias`) or hit a Playwright strict-mode
selector ambiguity against the real (larger, non-deterministic) library
(`player-focus`). This is a flow-tooling gap, not an observed app defect, and
would need `gogglebox-builder` to add a UAT-safe flow (e.g. select a specific,
known-unambiguous card by title, or `.first()` more narrowly) before the full
player-handoff can be visually proved against real data.
