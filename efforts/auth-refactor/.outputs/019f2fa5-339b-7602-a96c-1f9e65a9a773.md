# Visual proof — AuthRefactor acceptance criteria 2-6

## Summary

Ran the sbx stack from scratch in this session worktree
(`./sessions/auth-refactor.1783210783732`, compose project
`auth-refactor1783210783732`, separate from the orchestrator's main stack):
provisioned the sandbox Jellyfin (`sandbox-generate` + `sandbox-provision`),
brought up `server`/`client`, and — since the base `proxy` service hardcodes
host port 8080 and collided with the already-running main-stack proxy — added
a local, untracked `docker-compose.local-port.yml` override
(`services.proxy.ports: !override ["8180:8080"]`) purely to remap the host
port for this session's proxy container; no committed compose/source files
were touched. Ran `PROOF_FLOW=group-pin`, `PROOF_FLOW=group-alias`, and
`PROOF_FLOW=player-handoff` (the last with `-e PROOF_URL=http://proxy:8080`)
all under one `PROOF_RUN_ID=authproof-20260704T202336` so every screenshot
landed under `artifacts/authproof-20260704T202336/`. All three flow
invocations exited 0.

## Criteria confirmed

- **Criterion 2** (login accept/reject/persist) — **partially confirmed**.
  Valid access-token accept (both auto-login household and explicit visitor
  token submit), account-scoped picker routing, and logout-then-relogin
  (implying stored-token persistence and logout clearing it) are visually
  confirmed. Invalid-token rejection is NOT exercised by any current flow —
  no flow submits a deliberately wrong token, so there is no screenshot of
  the "clear error" UI for that path. Proof file:
  `efforts/auth-refactor/.proofs/019f2aa8-4931-7a1c-86ba-4785a38163c3.md`.
- **Criterion 3** (primary preselected/secondary visible/add-guest sizing) —
  **confirmed**. `group-alias-picker.png` (household: Alice+Bob preselected
  primaries, Carol+Dave visible unselected secondaries) and
  `group-pin-picker.png` (visitor: Dave-only secondary card, "+ Add guest"
  round control matching avatar diameter). Proof file:
  `efforts/auth-refactor/.proofs/019f2aa8-4933-796a-977a-7b9c756a7040.md`.
- **Criterion 4** (add-guest listing + PIN requirement + confirm gating) —
  **confirmed**. `group-pin-prompt.png` (PIN modal, confirm disabled until
  typed), `group-pin-wrong.png` (wrong PIN rejected in-place with a clear
  error, group not formed), `group-pin-success.png` (correct PIN unblocks
  confirmation and forms the group). Proof file:
  `efforts/auth-refactor/.proofs/019f2aa8-4934-7057-ac9a-c1ce09978cf1.md`.
- **Criterion 5** (mixed-group watch-progress warning) — **partially
  confirmed**. The warning modal's appearance (and its correct absence on the
  wrong-PIN path) is proved via a real, passing Playwright DOM-visibility
  assertion in `e2e/flows/group-pin.mjs`, gated on an actual server
  `verify-pins` response — but no screenshot exists of the "Shared watch
  progress" modal itself (the flow clicks through it without an intervening
  `shoot()` on the happy path). Recommend the builder add one screenshot call
  there. Proof file:
  `efforts/auth-refactor/.proofs/019f2aa8-4936-7965-b407-de265936e392.md`.
- **Criterion 6** (group selection + player handoff for token-authenticated
  accounts) — **confirmed for sandbox only**. `group-alias-created.png` shows
  a fully working token-authenticated group; `player-handoff-jellyfin-loggedin.png`
  and `player-handoff-jellyfin-playing.png` (plus flow log confirming
  `playbackStarted: true` with no stream errors) confirm the `/player`
  handoff. UAT/real-data coverage was NOT run — this session had no
  `.env.uat` / real Jellyfin available, so that half of the criterion is
  explicitly not covered rather than silently skipped. Proof file:
  `efforts/auth-refactor/.proofs/019f2aa8-4937-78c2-84fc-7c0eadca4372.md`.

## Limitations / follow-ups for the orchestrator

1. Criterion 2: needs a new (or extended) flow step that submits a
   deliberately invalid access token and screenshots the resulting `.error`
   message with the login form still present, to close the "rejects invalid
   tokens with a clear error" gap.
2. Criterion 5: needs one `shoot(page, 'group-pin-mixed-warning')` call in
   `e2e/flows/group-pin.mjs` right after the `.confirm-modal` becomes visible
   and before the "Confirm" click, so the warning text is directly
   screenshot-verifiable in a future pass.
3. Criterion 6: UAT/real-data proof is outstanding — needs a developer's real
   Jellyfin (`.env.uat`) configured and `./scripts/uat.sh` run through the
   same flows before this half can be marked proved.
4. I am read-only on source per role; did not edit any `e2e/` or `src/` file.
   The only file I added is an untracked, local-only
   `docker-compose.local-port.yml` (host port remap for this session's proxy
   container to avoid colliding with the orchestrator's already-running main
   stack) — it is not part of the effort's deliverable and can be deleted
   once this worktree's containers are torn down.
