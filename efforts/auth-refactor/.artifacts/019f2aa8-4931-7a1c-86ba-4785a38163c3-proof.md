# Proof: portal access-token login (accept/reject/persist)

## Criterion

Portal login accepts a valid access token, rejects invalid tokens with a clear
error, and stores the token locally until logout clears it.

## How proved (sandbox)

Sandbox stack (`./scripts/sbx.sh`), `PROOF_FLOW=group-pin`
(`e2e/flows/group-pin.mjs`), run under `PROOF_RUN_ID=authproof-prover-20260704`
in this session's own fresh worktree/volumes (sandbox re-provisioned from
scratch: `sandbox-generate` + `sandbox-provision`).

The flow logs out the auto-logged-in household session, defeats client
auto-login by patching `GET /api/session` (`portalAutoLoginEnabled: false`),
reloads to the logged-out state, then:

1. submits a deliberately bogus access token (`not-a-real-token`) and asserts
   the `POST /api/auth/login` response is a 4xx rejection, a visible `.error`
   message renders with non-empty text, the login form is still showing
   (nothing authenticated), and `localStorage.getItem('gogglebox.accessToken')`
   is NOT set — i.e. an invalid token is rejected and never persisted;
2. only then explicitly fills the single access-token field with the visitor
   account's valid token (`sbx-visitor-token`), gating success on an observed
   `POST /api/auth/login` response of `200` followed by the "Pick the group"
   picker rendering.

Flow log for the invalid-token step:
`[proof] group-pin: invalid-token login error = "Invalid access token"`, then
`PASS — invalid access token rejected (4xx + visible error), login form
persists, no token stored`. The flow exited 0 (`[proof] OK`).

## Verdict

Fully proved. All three clauses of the criterion are now visually confirmed
with durable, copied evidence: valid-token accept (picker renders per
account), invalid-token reject (4xx + visible error message + login form
persists + no token stored, screenshotted directly), and local persistence
until logout clears it (logout forces the login form to reappear). No
remaining gap.
