# Proof: mixed-group (non-primary) watch-progress warning

## Criterion

Continuing with a group that contains non-primary users shows a confirmation
warning that watch progress will be affected for every user in the group.

## How proved

Sandbox stack, `PROOF_FLOW=group-pin` (`e2e/flows/group-pin.mjs`), visitor
account, under `PROOF_RUN_ID=authproof-prover-20260704` (fresh sandbox
re-provisioned in this session's own worktree). Dave (secondary) + Carol
(tertiary guest, added via "+ Add guest") is a group with no primary members —
a mixed/non-primary group.

`e2e/flows/group-pin.mjs`'s `confirmPinsAndMixed()` now calls
`shoot(page, 'group-pin-mixed-warning')` right after `.confirm-modal` becomes
visible and before the "Confirm" click, producing a direct screenshot of the
open warning modal (previously this was only proved via a passing
element-visibility assertion, with no human-readable image).

## Evidence (copied into this effort's `.artifacts/`, not `./artifacts`)

- `efforts/auth-refactor/.artifacts/019f2fe2-9f64-7212-b76f-1289997defc7.png`
  (orig. `group-pin-mixed-warning.png`) — shows the "Pick the group" picker
  behind the modal (Dave + Carol both present, Carol added as guest) with the
  `.confirm-modal` open on top: heading "HEADS UP" / "Shared watch progress",
  body text "This group includes viewers beyond the default household set.
  Continuing will affect watch progress and watched states for ALL users in
  the group.", and "Cancel"/"Confirm" buttons — the exact warning text and
  UI the criterion requires, readable directly off the screenshot.
- Flow assertion (unchanged from prior pass): after a successful
  `POST /api/group/verify-pins`, the flow explicitly waits for `.confirm-modal`
  to become visible and fails loudly if it does not — this passed in this run
  — then clicks "Confirm" before the group POST fires and the picker is left.
- The flow also proves the NEGATIVE case is respected: when the wrong PIN was
  submitted, the flow explicitly asserts the mixed-group modal did **not**
  appear, confirming the warning is gated correctly on successful PIN
  verification rather than appearing unconditionally (see
  `efforts/auth-refactor/.artifacts/019f2fc4-08aa-783a-b276-4a74853259a1.png`,
  the wrong-PIN screenshot from the prior pass, which shows the picker/PIN
  modal — NOT the mixed-group modal — still open).

## Verdict

Fully proved. The warning's presence, exact copy, and gating (shown only
after successful PIN verification, never on a wrong PIN) are all now
confirmed both by a passing Playwright assertion tied to real server state
and by a screenshot a human can read directly showing the "Shared watch
progress" modal and its full warning text. No remaining gap.
