# Proof — AC7: `publish` needs both `test` and `e2e`; failure-skip (partially proven)

## Criterion

`publish`'s `needs:` includes both `test` and `e2e`, and that a
deliberately-broken flow (temporary local/dry-run repro, reverted after
proving) causes `publish` to be skipped exactly as a deliberately-broken
unit test does today, while `release.yml`'s promotion behavior is unchanged
(no retest, no rebuild).

## Evidence reviewed

- `.github/workflows/publish.yml` (line 167) reads directly:

  ```yaml
  publish:
    needs: [test, e2e]
  ```

  This satisfies the static `needs:` half of the criterion by direct
  inspection — confirmed today, in the repo, not just asserted by a builder.

- `release.yml` was not touched by this effort (only `publish.yml` appears
  in the two builder handoffs' file lists, and the file's promotion
  behavior — promoting an already-built, already-tested image without
  rebuilding/retesting — is unaffected by construction, since no PR in this
  effort edits it).

- The approval-handoff prompt asserts a live CI proof: the same
  `scratch/e2e-gate-live-proof` run (28748139115) showed the `publish` job
  as skipped (`-`, 0s runtime) once `e2e` failed, demonstrating
  `needs: [test, e2e]` skip-propagation in a real run, and that no image was
  built or pushed to GHCR.

## Why the live-CI half is NOT sufficient proof

Same evidentiary gap as AC4: the "publish shows skipped" observation is
asserted only in the handoff prompt's narrative text. There is no proof
artifact (e.g. a saved screenshot of the Actions run's job-graph view, or a
`gh api` JSON dump of the run's jobs list showing `"conclusion": "skipped"`
for `publish`) copied into `efforts/release-e2e-gate/.proofs/`. This agent
has no network/`gh` access to independently re-verify the cited run URL, and
per the approval rules a claim "only asserted in chat" must be left
unchecked rather than accepted on the orchestrator's word alone, however
credible.

## Verdict

The static-`needs:`-list fact (`publish: needs: [test, e2e]`, confirmed by
direct file read today) is real and durable, but the criterion as written
also requires proof of the actual skip-behavior from a deliberately-broken
flow, which depends on the same unconfirmed live-CI artifact as AC4. Since
the criterion is a single conjunctive statement (`needs:` correctness AND
the failure-skip demonstration), it is left unchecked as a whole. Missing:
a durable artifact (screenshot or API-response dump) placed in `.proofs/`
showing the `publish` job's `skipped` conclusion on run 28748139115 (or an
equivalent reproducible dry-run), analogous to what AC4 needs.
