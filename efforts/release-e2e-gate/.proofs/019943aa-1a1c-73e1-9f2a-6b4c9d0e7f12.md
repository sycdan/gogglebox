# Proof — AC2: `fail()` exit-code propagation

## Criterion

A flow's existing `fail()` call (assertion failure) is confirmed to
propagate as a non-zero process exit from `e2e/run.mjs` today, with no new
error-propagation code required — documented as a verified fact, not an
assumption.

## Evidence

Source: `efforts/release-e2e-gate/.outputs/019f301a-1c2b-75ac-be64-a8f5e4d4ff49.md`
(gogglebox-builder), cross-checked against `e2e/lib/harness.mjs`'s `fail()`:

```js
export function fail(message, error) {
  console.error(`[proof] FAIL: ${message}`);
  if (error) console.error(error instanceof Error ? error.stack : error);
  process.exit(1);
}
```

`fail()` calls `process.exit(1)` directly (no throw, no reliance on an
uncaught-exception handler) — the moment any flow calls it, the process
terminates immediately with exit code 1.

Builder additionally verified this empirically, not just by reading source:
a scratch script imported `fail` from `e2e/lib/harness.mjs` and called it
directly via `docker compose run --rm check node /app/e2e/ac2-check.scratch.mjs`
(temporary, deleted after the check). Output:

```
[ac2-check] about to call fail()
[proof] FAIL: deliberate AC2 verification failure

EXIT_CODE=1
```

The line after `fail()` never printed; `$?` was `1`. Separately, both the
AC1 Run 2 all-flows failure and a standalone `PROOF_FLOW=player-focus`
invocation against the same broken sandbox state both returned `docker
compose run --rm ... proof` exit code 1, captured directly via `$?`.

No error-propagation plumbing was added — `fail()` was already sufficient in
both single-flow and all-flows mode, confirmed as a pre-existing fact.

## Verdict

Sufficient. Checked.
