# Proof — AC4: artifact upload on failure (NOT sufficiently proven)

## Criterion

The `e2e` job uploads `./artifacts/**` as a build artifact when the flow run
fails, and that this artifact is confirmed retrievable from a real (or
dry-run) Actions run showing the failure screenshots.

## Evidence reviewed

- `.github/workflows/publish.yml` (lines 149-155) does contain the step:

  ```yaml
  - name: Upload artifacts on failure
    if: failure()
    uses: actions/upload-artifact@v4
    with:
      name: e2e-artifacts
      path: ./artifacts/**
      if-no-files-found: ignore
  ```

  This satisfies the YAML-authoring half of the criterion by direct
  inspection.

- The builder's own output
  (`efforts/release-e2e-gate/.outputs/019f3086-7177-7c73-8971-f0c4f3341e04.md`)
  explicitly states this step was NOT empirically confirmed against a real
  Actions run in that handoff, and flags it as the concrete gap.

- The approval-handoff prompt
  (`efforts/release-e2e-gate/.prompts/019f333d-cec9-778f-8e73-523e08786076.md`)
  asserts that the orchestrator subsequently ran a live CI proof: branch
  `scratch/e2e-gate-live-proof` cut from `main`, `group-alias.mjs`
  deliberately broken, pushed, dispatched via `gh workflow run publish.yml`,
  producing a real run at
  `https://github.com/sycdan/gogglebox/actions/runs/28748139115` whose `e2e`
  job failed as expected, whose "Upload artifacts on failure" step ran
  green, and whose resulting `e2e-artifacts` artifact (101615 bytes) was
  downloaded and confirmed to contain a real screenshot at
  `2026-07-05T16-59-29-372Z-all/group-alias/group-alias-02-authenticated.png`.

## Why this is NOT sufficient proof

This session verified the following facts directly in the working tree:

- `e2e/flows/group-alias.mjs` contains no `DELIBERATE`/deliberate-failure
  marker today — consistent with the claim that the scratch branch was
  reverted/discarded, but this alone does not prove the described run
  happened.
- No local `scratch/e2e-gate-live-proof` git ref exists (checked
  `.git/refs/heads/scratch/*` — none found), consistent with (but not proof
  of) "deleted after use."
- Critically: the specific downloaded artifact file the prompt cites,
  `2026-07-05T16-59-29-372Z-all/group-alias/group-alias-02-authenticated.png`,
  does **not exist anywhere in this worktree** — not under
  `efforts/release-e2e-gate/.proofs/` (the only place a screenshot proof can
  durably live per the effort-approval rules) and not even under the
  gitignored, ephemeral `./artifacts/` directory where a locally-run proof
  would land. The claim of a downloaded artifact is asserted only in the
  handoff prompt's chat-style narrative text, with no file evidence to
  back it and nothing copied into `.proofs/`.
- Per the approval rules: "A proof doc whose cited screenshot path resolves
  outside its own `.proofs/` directory ... is insufficient" and "If proof is
  ... only asserted in chat, leave the criterion unchecked." This evidence
  is asserted in chat/prompt text only, with the cited artifact absent even
  from the ephemeral location, let alone copied into `.proofs/`.
- A GitHub Actions run URL that this approval pass cannot independently
  fetch (no network/`gh` tool access available to this agent) is not, by
  itself, durable proof — even if the run genuinely occurred, the artifact
  it produced must be copied into `.proofs/` (per the agent guide: "any
  screenshot evidence must be copied from root `./artifacts` into the
  effort's `.proofs/` directory") for it to survive session teardown and be
  checkable by a future reader.

## Verdict

Not sufficient. Left unchecked. Missing: the actual downloaded artifact
screenshot(s) from run 28748139115 (or an equivalent real/dry-run Actions
run) copied into `efforts/release-e2e-gate/.proofs/`, so the retrievability
claim is durable evidence rather than a chat assertion.
