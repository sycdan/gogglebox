# Proof — AC5: sandbox teardown regardless of pass/fail

## Criterion

The `e2e` job tears down the sandbox stack after it finishes regardless of
pass/fail (e.g. via `if: always()`), leaving no dangling containers on the
runner.

## Evidence

Source: `efforts/release-e2e-gate/.artifacts/019f3086-7177-7c73-8971-f0c4f3341e04-output.md`
(gogglebox-builder), cross-checked directly against
`.github/workflows/publish.yml`:

```yaml
- name: Tear down sandbox stack
  if: always()
  run: docker compose -f docker-compose.yml -f docker-compose.sbx.yml down -v --remove-orphans
```

`if: always()` present, confirmed by direct read of the workflow file
(line 159-161).

Builder locally validated the teardown for real: after running the full
bootstrap sequence in a fresh worktree (jellyfin-sandbox up, generate,
provision, reset, stack up, healthcheck confirmed healthy),
`docker compose -f docker-compose.yml -f docker-compose.sbx.yml down -v
--remove-orphans` cleanly removed every container/volume/network for that
session's own compose project (`release-e2e-gate1783225546089-*`), verified
zero containers remained afterward (`docker ps -a --filter
name=release-e2e-gate1783225546089` empty), while leaving a separate,
unrelated `gogglebox-*`-prefixed dev stack running elsewhere on the host
completely untouched — confirming the teardown is scoped to its own compose
project, not globally destructive.

The live-CI proof pass (see AC4/AC7 evidence below, orchestrator-executed
run https://github.com/sycdan/gogglebox/actions/runs/28748139115) also shows
the "Tear down sandbox stack" step ran green despite the deliberate `e2e`
failure, reconfirming `if: always()` fires on failure in a real Actions run
as well as locally.

## Verdict

Sufficient. Checked.
