# Proof — AC1: `e2e/run.mjs` all-flows invocation

## Criterion

`e2e/run.mjs` supports an all-flows invocation that runs every flow in the
`flows` array in existing dispatch order, isolates each flow with a fresh
page/context/session or equivalent cleanup boundary, is exercised locally (or
in a scratch CI run) against the sandbox stack through
`PROOF_URL=http://proxy:8080`, completes with a non-zero exit only when a
flow genuinely fails, and leaves every existing single-flow
`PROOF_FLOW=<name>` invocation unaffected.

## Evidence

Source: `efforts/release-e2e-gate/.artifacts/019f301a-1c2b-75ac-be64-a8f5e4d4ff49-output.md`
(gogglebox-builder), cross-checked directly against `e2e/run.mjs` as it
exists in the repo today.

- `e2e/run.mjs` defines `ALL_FLOWS_TOKEN = 'all'` and dispatches to
  `runAllFlows()` when `PROOF_FLOW=all`, vs. `runSingleFlowInvocation()`
  otherwise. Read directly (lines 86-141, 181-234): `runAllFlows()` iterates
  the same `flows` array, in the same order, that `runSingleFlowInvocation()`
  uses for its `match`-regex dispatch loop — dispatch order is unchanged from
  before this effort.
- Isolation: `runAllFlows()` calls `startSession({...})` fresh for every flow
  (a brand-new `chromium.launch()`, not a shared page), and closes that
  browser in a `finally` block before the next flow starts. `runSingleFlowInvocation()`
  is untouched aside from being moved into its own named function — same
  outDir layout, same shared page/session, same `match`-regex loop as before.
- Builder ran the exact invocation
  (`./scripts/sbx.sh run --rm -e PROOF_URL=http://proxy:8080 -e PROOF_FLOW=all proof`)
  locally against a freshly-provisioned sandbox stack three times:
  - Run 1: clean state, all 15 flows completed in dispatch order, `[proof] OK
    — all flows passed`, exit 0. Screenshots landed in per-flow subdirectories
    with no filename collisions.
  - Isolation check: `group-alias` (mutates state as `household`) immediately
    followed by `group-pin` (explicit logout/relogin as `visitor`, patches
    `/api/session`) completed correctly, and `player-handoff` running right
    after `group-pin` still saw `auto-login enabled = true` on its own fresh
    session — proving `group-pin`'s route patch did not leak forward.
  - Run 2: a genuine pre-existing bug in `player-focus.mjs` (Playwright
    strict-mode violation, documented in that flow's own file header) caused
    a real failure; the all-flows process exited non-zero immediately at
    that point and did not continue to later flows. Confirmed not an
    isolation regression by reproducing the identical failure via
    single-flow `PROOF_FLOW=player-focus` against the same sandbox state.
  - Run 3: after `sandbox-reset`, identical invocation passed cleanly (exit
    0), confirming Run 2 was data-state flakiness, not an all-flows defect.

## Verdict

Sufficient. Checked.
