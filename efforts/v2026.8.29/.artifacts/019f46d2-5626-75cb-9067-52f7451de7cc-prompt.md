## The design (canonical decisions — encode these into the specs)

### Product thesis

Gogglebox optimizes **time-to-play for a room**, not engagement. North-star
interaction: log in → "Press Start" → the top recommendation plays. North-star
metrics: time-to-play (login → playback start) and deck-advance/new-deck counts
per session; both recorded as facts. Deliberately anti-engagement: no autoplay
dark patterns, no infinite browse. Every layer must degrade gracefully to a
zero-config default (fresh deploy with no answered questions still gets a good
rail). Explainability is the trust engine: never surface an item without a
renderable reason.

### Architecture: facts → signals → aggregation → deterministic score → deck

1. **Fact store**: append-only JSONL event log in the state dir. Shape:
   `{ ts, userId, itemId, kind, value, provenance }`. Fact kinds include:
   `seen`, `liked`, `rewatch`, `completed`, `skipped`, `ignored`,
   `started-blind` (played via Press Start), `backed-out`, `not-tonight`,
   `elicited-*`, plus session telemetry (deck advances, new-deck deals).
   `provenance` distinguishes `own-pad` (self-reported via bound controller)
   from `room-reported`. New data source = new fact kind, no schema migration.
2. **Signal plugins**: each signal implements
   `score(item, partyContext) -> { value: 0..1, reason?: string } | null`
   (null = abstain; weights renormalize over non-abstaining signals so missing
   data never punishes an item). Signals are registered in one registry;
   adding a signal = one module + one registry entry + one config weight.
   Initial signal set: recently-watched-by-another-member, liked-by-trusted-user
   (trust explicitly declared v1, inferred later), highly-rated, newly-added
   (rank high — owner added it on purpose; a uniquely strong home-library
   signal), most-watched, least-watched, fits-time-budget.
3. **Party aggregation**: each signal declares its aggregator across present
   members — **least-misery (min)** for veto-shaped signals (kid-safe,
   hated-genre, disliked), **mean** for taste-shaped signals. Negative facts
   (disliked/ignored/skipped/not-tonight) weigh roughly 2-3x positives: group
   satisfaction tracks the least-happy member.
4. **Hard filters** (pre-score, not signals): kind (movie/show/both), genre,
   kids-only, hard time-budget cutoff. Time budget also reappears as a soft
   fit signal. Budget source: direct input v1; inferred from playback-history
   session rhythms v2.
5. **Deck**: score all candidates deterministically → top-K (~20) →
   score-weighted shuffle with a per-session seed → deal "tonight's deck"
   (~7). Deck is finite and shows position ("3 of 7"). Advancing the deck
   replaces the old reroll concept entirely. Exhausting it offers an explicit
   "deal new deck" (fresh sample excluding seen; logged as a strong
   dissatisfaction fact). Same session seed = reload-stable. This resolves the
   old determinism-vs-reroll contradiction.
6. **Explanation**: each shown item renders its top ~2 contributing reasons
   (by weight x value) as human sentences.
7. **No long-running ranking daemon**: scoring is on-demand (household scale =
   milliseconds). Only expensive Jellyfin aggregates (watch counts, watched
   unions) may be cached/precomputed. Elicitation answers affect the very next
   request.
8. **Rewatch must be possible**: the current watched-union exclusion makes
   comfort rewatches unrecommendable; the new design must allow a
   watched-by-all rewatch path (e.g. via the rewatch/liked facts), since it is
   the group's highest-floor pick.
9. **Eval discipline**: an offline replay harness (hold out each party's last
   watch, measure hit-rate@10 over the fact history; also track Start-press
   acceptance) runs deterministically against the sandbox. Rule: no new signal
   without a named failure it fixes and an eval delta showing it.

### Surface: proposal deck (hero + on-deck strip), not a capacity-paged rail

- **Hero card** (~70-80% of screen): art, title, runtime, legible-from-couch
  reasons, member vote pips. The algorithm "makes its argument" here.
- **On-deck strip**: next 3-5 deck items as small thumbnails (art + title
  only) for recognition ("that one!") and calibration. On a phone the strip
  collapses to pure one-at-a-time. This REPLACES the old responsive
  capacity-count paging AC — hero is always exactly one; the strip is simple
  overflow.
- **Resume slot**: persistent, fixed placement, outside the deck flow — if the
  party has in-progress items, resume is always one action away, never buried.
  Mixed card semantics must be coherent: episode cards (progress, Resume,
  scoped ignore) vs title cards (Start).
- **Correction loop**: every card carries one-press "not for us" (permanent-ish,
  the existing ignore) and "not tonight" (session-soft). Passive correction
  beats elicitation for effort-to-value and must be effortless.

### Input model: controller-first spatial focus

- W3C Gamepad API; Xbox-style pads get `"standard"` mapping. Poll via
  requestAnimationFrame with edge detection and key-repeat for held d-pad;
  "press any button" gate satisfies Chrome's gesture requirement.
- One **spatial focus engine**, two adapters: keyboard arrows/Enter and
  gamepad. Mouse/touch remain as progressive enhancement. Controller-first
  bans hover-dependent UI and text-input-dependent flows (elicitation answers
  are button presses, never text; budget input is a stepper).
- Button map: d-pad = deck nav, **A** = play focused, **B** = back, **X** =
  not-for-us/veto, **Start** = play the hero immediately, LB/RB = page where
  applicable.
- **One driver, N voters**: navigation is single shared focus (last-active pad
  drives); parallel multi-pad input only at explicit vote/answer moments.
- Multi-pad v1.5 (spec as later-ordered subeffort or clearly-deferred goals):
  claim-to-join party formation ("Alice — press A to join", pad index bound to
  viewer, facts gain `own-pad` provenance), parallel elicitation answers,
  attributed vetoes.
- Handoff seam: gamepad support ends at the `/player` Jellyfin-web boundary;
  Jellyfin-web's own gamepad behavior is unverified. Pads are expected to go
  down after choosing anyway; playback control may be keyboard/mouse/remote.
  State this seam explicitly in the input spec.

### Search (revised)

Keep the server-side Jellyfin search endpoint and inter-show episode results.
DROP the "search filters the rail's result set" constraint — it contradicted
returning global Jellyfin results. New shape: search results REPLACE the deck
surface in a dense grid **exempt from deck/hero layout**, same single surface,
clear escape back to the deck, clear empty state. Search is known-item/high-
intent and phone-shaped; it is de-emphasized in the controller context.

### Elicitation

Login-time micro-question, maximum one, one-press answers, skippable forever
without nagging. Question selection: highest candidacy x fewest facts for
present members. Answers land as facts and affect the next request. Ordered
LAST among subefforts (needs facts, deck, and correction loop live first).
