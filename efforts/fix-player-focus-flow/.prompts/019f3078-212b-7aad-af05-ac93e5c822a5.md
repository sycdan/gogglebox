## Confirmed background (from a completed release-e2e-gate handoff)

`efforts/release-e2e-gate/.outputs/019f301a-1c2b-75ac-be64-a8f5e4d4ff49.md`
documents a confirmed, pre-existing bug in `e2e/flows/player-focus.mjs`:

- It locates the Play/Resume button with a page-wide `.or()` selector:
  ```js
  const playButton = page
    .locator('.media-card button', { hasText: /^Play$/ })
    .first()
    .or(page.locator('.media-card button', { hasText: /^Resume$/ }).first());
  ```
  This is ambiguous (Playwright strict-mode violation) once more than one
  `.media-card` on the page has a visible Play or Resume button — e.g. after
  `mark-all-watched` runs, or against a larger seeded/real library where
  several cards are simultaneously resumable.
- Reproduced twice with identical failure and exit code: once inside
  `PROOF_FLOW=all` (all-flows mode, `player-focus` ran after other flows had
  mutated watched/continue-watching state) and once standalone via
  `PROOF_FLOW=player-focus` against the exact same sandbox state. Identical
  failure message (`FAIL: player-focus: found no Play/Resume button to open
  the modal`) and identical non-zero exit both times — this confirms it is a
  flow-authoring bug in `player-focus.mjs` itself, not an all-flows
  isolation/harness regression.
- `e2e/flows/player-uat.mjs`'s own file header comment (lines ~12-16) already
  documents this exact class of bug and the fix pattern it adopted to avoid
  it: instead of a page-wide `.or()`, it iterates `.media-card` elements one
  at a time (`page.locator('.media-card').nth(index)`) and scopes the
  Play/Resume button search to a SINGLE chosen card
  (`card.locator('button', { hasText: /^(Play|Resume)$/ }).first()`),
  picking the first card that has one. See `e2e/flows/player-uat.mjs` lines
  ~85-108 for the exact working pattern.
- `player-focus.mjs`'s job (see its own file header) is to prove the player
  modal is keyboard-safe: opening it focuses the dialog (not the opener
  button), locks page scroll, mounts the Jellyfin iframe, Space does not
  re-trigger the opener, and Escape closes the modal. None of that behavior
  is in question — only the button-locating step that runs before any of it.

## Goal of this effort

Make `player-focus.mjs` locate its target Play/Resume button unambiguously —
mirroring `player-uat.mjs`'s per-card-scoped approach rather than a
page-wide `.or()` — so the flow passes reliably regardless of how many
watched/resumable cards are present on the page, verified against the sandbox
stack (both in isolation and after state that produces multiple
simultaneously-resumable cards, e.g. after `mark-all-watched`).
