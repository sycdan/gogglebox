## Show Detail Browser — builder output

### Status: pass (static verification green; visual proof still needed)

### What was implemented

**Server (`src/server/jellyfin.ts`, `src/server/server.ts`)**

- `JellyfinClient.listEpisodes(seriesId, searchTerm?)` now accepts an optional
  keyword. `ParentId=seriesId` is set on every call (search or not), so
  Jellyfin's `SearchTerm` filter only narrows WITHIN that series — it can never
  surface another show's episodes (AC4).
- New `withEpisodeViewerWatchedState()` helper (module-level + the identical
  copy inside `createApp`, matching this file's existing duplication pattern)
  attaches every active party viewer's live `Played` state to each episode row,
  reusing the same per-user `getItemPlayedState` read the continue-watching
  cards already use — read-only, no new mutation path (AC3, respects the
  "no watched-state editing" nongoal).
- `GET /api/shows/:seriesId/episodes` now accepts `?q=<term>` and returns
  `{ items: EpisodeItemWithWatched[] }`, where each item carries a
  `viewerWatched: ViewerWatchedState[]`.
- New exported type `EpisodeItemWithWatched` in `server.ts`.

**Client (`src/client/App.tsx`, `src/client/styles.css`)**

- `openEpisodes` renamed/generalized to `openShowDetail(series: { id, name })`
  and `closeShowDetail()`. The narrower `{ id, name }` shape (rather than the
  full `LibraryItem`) lets BOTH library cards (recs/search rails) and
  continue-watching cards (which only carry `seriesId`/`seriesName`) open the
  same modal.
- **AC1**: Show titles are now clickable everywhere they appear:
  - `MediaCard`'s `<h3>` title is a button for shows (movies keep a plain
    heading — no show detail exists for them). The existing "Episodes" button
    remains as a secondary entry point.
  - The continue-watching show card's series name (in the `.meta` line) is now
    a button too.
  - Opening the modal is a purely local state change (`selectedSeries`/
    `episodes`/etc.) — it does not touch `session`, `recommendations`,
    `searchResults`, `continueWatching`, or any other page-level state, so the
    page behind it is untouched, matching the existing modal pattern used by
    the player/ignored modals.
  - Added `role="dialog"`, `aria-modal="true"`, `aria-label`, a focus-on-open
    effect (mirrors the existing player-modal pattern), and Escape-to-close.
- **AC2**: Season grouping/filter. Season numbers are derived from the
  currently-loaded episode list and rendered as a button row ("All seasons" +
  one per season); selecting a season filters the visible episode list
  client-side. The season row narrows along with an active keyword search
  (buttons only show seasons still present in the filtered results).
- **AC3**: Each visible episode row renders a row of per-viewer pills (reusing
  the existing `viewer-pill` look from continue-watching cards) showing
  watched/unwatched per active-party viewer. These pills are **display-only**
  (`viewer-pill-static`, no `onClick`) — no new watched-state editing was
  added, per the effort's nongoal.
- **AC4**: A "Search episodes in `<Show>`" field inside the modal, debounced
  (500ms) and scoped via `?q=` to `GET /api/shows/:seriesId/episodes`. This is
  a separate `episodeSearchQuery` state/effect from the top-level library
  search (`searchQuery`), so it can never become — or be confused with — a
  global/discovery-rail search. Clearing the field re-fetches the full
  unfiltered list.
- New CSS: `.show-detail-modal`, `.link-title` / `.link-title-inline` (title
  rendered as an unstyled, keyboard-accessible button), `.season-filter-row`,
  `.viewer-pill-static`.

### Deviations from spec / assumptions

- The existing episode-modal "Mark watched" button (whole-episode mark, all
  active viewers at once) was already present before this effort and is left
  as-is — it is pre-existing behavior, not something this effort added. The
  new per-viewer pills added for AC3 are intentionally read-only so this
  effort does not introduce any NEW watched-state editing surface, per the
  nongoal ("do not edit watched state from this modal unless a later effort
  specifies it").
- Season filter is a client-side filter over the already-fetched episode list
  rather than a separate `?season=` server query param — simpler, and the
  full per-series episode list is already a single cheap Jellyfin call (same
  approach the pre-existing code used before this effort).
- Assumed "clicking a show title anywhere it appears" covers: media cards
  (recommendations + search rails) and continue-watching cards. There is no
  other place a show title renders in the current UI (no global nav/breadcrumbs
  reference a show name).
