1. **No `derivePartyKey` golden-value regression test.** The source comment in
   `src/server/partyKey.ts` states the `PARTY_NAMESPACE` UUIDv5 namespace
   constant "must NEVER change" because it's baked into every previously
   persisted key and minted Jellyfin username. The builder's output claims
   this was preserved unchanged from the old `GROUP_NAMESPACE`. However,
   `partyKey.test.ts` only asserts internal self-consistency (determinism,
   order-independence, dedup, hex format) — none of it pins a known/expected
   output hash for a fixed input. A future edit that accidentally changes the
   namespace constant would still pass every existing test. Recommend adding
   one test asserting `derivePartyKey(['user-a','user-b'])` equals a specific
   hard-coded 32-char hex value computed today, to catch that class of
   regression.

   `GET /api/session` / `GET /api/parties` return both old and new field names
   with identical values (`activeGroupAlias`/`activePartyAlias`,

   automated test, unit or otherwise. This appears to be a pre-existing gap in
   the codebase's test strategy (no route-level tests exist for any endpoint,
   not just the new party ones), not something newly introduced by this
   effort, but it means the route-alias/dual-field claim currently rests
   entirely on the builder's manual curl checks against a live sandbox (not
   reproducible in CI) plus visual inspection of `server.ts`'s route
   registrations (lines around `app.post('/api/group/verify-pins', ...)`,

   `app.post('/api/group/clear', ...)`, all pointing at the same handler
   functions as the `/api/party*` routes). Recommend either a lightweight
   supertest-based route test, or at minimum documenting in the effort's proof
