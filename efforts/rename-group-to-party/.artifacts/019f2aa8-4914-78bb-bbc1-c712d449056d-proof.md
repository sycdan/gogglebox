# AC2 proof — party-oriented API/state, with group-oriented compatibility

## Canonical `/api/party*` routes, wired identically to their `/api/group*` aliases

`src/server/server.ts` registers each canonical route immediately followed by
its pre-rename alias, both pointed at the exact same handler function:

```
379:app.post('/api/party/verify-pins', requireAuth, handleVerifyPartyPins);
381:app.post('/api/group/verify-pins', requireAuth, handleVerifyPartyPins);
...
429:app.post('/api/party', requireAuth, handleCreateParty);
431:app.post('/api/group', requireAuth, handleCreateParty);
...
456:app.get('/api/parties', requireAuth, handleListParties);
458:app.get('/api/groups', requireAuth, handleListParties);
...
465:app.post('/api/party/clear', requireAuth, handleClearParty);
467:app.post('/api/group/clear', requireAuth, handleClearParty);
```

Each alias line carries an explicit `// Pre-rename compatibility alias —
identical behavior.` comment (`server.ts:380`, `430`, `457`, `466`).

## Dual-named response fields

- `GET /api/session` (`server.ts:266-291`) computes one `partyAlias` value
  once (`activePartyAlias(req, ...)`, `server.ts:272`) and returns it under
  **both** names, with the compatibility field commented as never diverging:

  ```
  286:    activePartyAlias: partyAlias,
  287:    // Pre-rename compatibility field (identical value) for any client still
  288:    // reading the old name. Never diverges from activePartyAlias above.
  289:    activeGroupAlias: partyAlias,
  ```

- `handleListParties` (`server.ts:436-455`) returns both shapes from one
  computed list:

  ```
  449:  res.json({
  450:    parties,
  451:    // Pre-rename compatibility field: the same list, shaped with `groupKey`
  452:    // instead of `partyKey`, for any client still reading the old field names.
  453:    groups: parties.map(({ partyKey, ...rest }) => ({ groupKey: partyKey, ...rest })),
  454:  });
  ```

## Client state uses the new names exclusively (server keeps both)

`src/client/App.tsx`:
- `SessionResponse.activePartyAlias` (line 157) — the client only reads the
  new field.
- `SavedParty` interface (`164-165`) is keyed by `partyKey`, with a comment
  (`161-163`) explicitly noting parties "were formerly called 'groups' — see
  `/api/parties` (and its `/api/groups` compatibility alias)".
- The client calls `/api/parties` (421), `/api/party/verify-pins` (851),
  `/api/party` (918), `/api/party/clear` (947) exclusively — the shipped
  client never calls `/api/group*` itself, but the server keeps those routes
  live for any other consumer (documented in the builder's implementation
  summary,
  `efforts/rename-group-to-party/.artifacts/019f33b8-bcab-7de3-9b20-ccc8bf0dfc57-output.md`).

## Live confirmation against a real sandbox (this proof phase)

Brought up a fresh sandbox stack (`./scripts/sbx.sh up -d jellyfin-sandbox` →
`sandbox-generate` → `sandbox-provision` → `up -d`) and drove the running
server directly with `curl` (household token `sbx-household-token`, viewers
Alice=`1a53a2e...c95`, Bob=`2cf8ee4a...1a`):

1. **Session before any party exists** — both alias fields present and
   agreeing (both `null`):
   ```json
   {"activeViewerIds":[],"activePartyAlias":null,"activeGroupAlias":null}
   ```

2. **`POST /api/group` (legacy route) forms Alice+Bob** →
   `{"ok":true,"activeViewerIds":["1a53a2e...c95","2cf8ee4a...1a"]}`

3. **`GET /api/session` afterward** — both alias fields now agree on the
   same value, proving the legacy-route write and the new-field read are the
   same underlying state:
   ```json
   "activeViewerIds":["1a53a2e...c95","2cf8ee4a...1a"],
   "activePartyAlias":"Alice + Bob",
   "activeGroupAlias":"Alice + Bob"
   ```

4. **`GET /api/parties` (new route)** returns dual-shaped list, `partyKey`
   and `groupKey` carrying the identical hash for the identical members:
   ```json
   {"parties":[{"partyKey":"16a4a66ebeb65df0b495bfa929fb5cee","alias":"Alice + Bob","memberIds":["1a53a2e...c95","2cf8ee4a...1a"],"memberNames":["Alice","Bob"]}],
    "groups":[{"groupKey":"16a4a66ebeb65df0b495bfa929fb5cee","alias":"Alice + Bob","memberIds":["1a53a2e...c95","2cf8ee4a...1a"],"memberNames":["Alice","Bob"]}]}
   ```

5. **`GET /api/groups` (legacy route)** returned a byte-for-byte identical
   body to step 4's `/api/parties` response (same `parties`+`groups` dual
   shape) — confirming both routes hit the same handler
   (`handleListParties`), not just similarly-shaped independent code paths.

6. **`POST /api/party/clear` (new route)** → `{"ok":true,"activeViewerIds":[]}`,
   then **`POST /api/party` (new route)** reformed Alice+Bob successfully,
   then **`POST /api/group/clear` (legacy route)** →
   `{"ok":true,"activeViewerIds":[]}` and a follow-up `GET /api/session`
   confirmed `activeViewerIds:[]`, `activePartyAlias:null`,
   `activeGroupAlias:null` — new and legacy clear routes both fully reset the
   active party.

7. **Pin verification parity**: logged in as the `visitor` account (Carol is
   a tertiary/pin-gated guest, pin `5678`). `POST /api/group/verify-pins`
   (legacy route) with a wrong pin (`0000`) returned HTTP `403`. `POST
   /api/party/verify-pins` (new route) with the correct pin (`5678`)
   returned `{"ok":true}` — proving both routes are wired to the same
   `handleVerifyPartyPins` validation logic (reject wrong pin, accept
   correct pin), not divergent implementations.

Sandbox was torn down (`./scripts/sbx.sh down`) after this verification;
these are fresh outputs from this proof phase's own run, not reused from an
earlier phase.
