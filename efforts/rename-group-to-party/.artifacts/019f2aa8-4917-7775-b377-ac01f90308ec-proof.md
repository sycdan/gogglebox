# AC4 proof — docs/sandbox/automation consistently use "party", with legacy compat notes

(`feat(rename-group-to-party): rename group to party terminology across app,

## `README.md`

- Positioning language: "the best possible experience choosing and watching
  something as a **party**" (was "as a group"); "Gogglebox sits in front of
  it as a focused, **party-aware** layer" (was "group-aware").
- Explicit compatibility note added in the architecture section: "parties
  are formed live in the UI (a party is a Jellyfin user created on demand).
  **Parties were formerly called "groups" — the server still accepts the
  old `/api/group*` routes and response fields as compatibility aliases (see
  `src/server/server.ts`).**"
- Tier docs: "the configured user PIN whenever they are added to a **party**
  for that account" (was "to a group").
- Roadmap section: removed the now-stale "renaming the visible 'group'
  concept to 'party'" backlog bullet (the rename is done, not upcoming) —
  current roadmap bullets read "authentication, persistence, show-detail
  browsing, and the `v2026.8.29` 'Judgement Day' discovery work."

Confirmed present on the current worktree (not just in the historical diff)
via `grep -in group README.md`: the only remaining hits are the
compatibility-note sentence above and the PIN-tier sentence's surrounding
context — no stray/leftover "group" language describing current behavior as
primary.

## UAT stack — no group/party language to migrate (confirmed, not a gap)

Checked `docker-compose.uat.yml` and `scripts/uat.sh` directly on the
current worktree:

```
grep -in "group" docker-compose.uat.yml scripts/uat.sh scripts/sbx.sh
(no matches)
grep -in "party\|parties" docker-compose.uat.yml
(no matches)
```

Neither file ever used group/party terminology — they're generic
env-layering/service-pointing wiring (real Jellyfin creds via
`.env.uat`/`config.uat.json`), so there was nothing to rename and no
compatibility note is needed there. `tools/sandbox/README.md` (already
covered above) is the doc that carries the sandbox-specific party
terminology and its compat notes; the UAT flow reuses the same server/client
code (already party-oriented per AC2/AC3) without any UAT-specific
group/party prose.

## Live confirmation the running app/docs are internally consistent (this proof phase)

As part of gathering AC2 evidence in this same proof phase, a fresh sandbox
stack was provisioned from these exact doc-described commands
(`./scripts/sbx.sh up -d jellyfin-sandbox`, `run --rm sandbox-generate`,
`run --rm sandbox-provision`, `up -d`) and the resulting `.env.sbx` +
`config.sbx.json` matched `tools/sandbox/README.md`'s description exactly:
`household`/`sbx-household-token` (Alice+Bob primary, Carol+Dave secondary)
and `visitor`/`sbx-visitor-token` (Dave secondary, Carol tertiary/pin-gated)
— i.e. the README's party-oriented description of the sandbox accounts is
accurate to what the committed automation actually produces today, not
stale prose.
