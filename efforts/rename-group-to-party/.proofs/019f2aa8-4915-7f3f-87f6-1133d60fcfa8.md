# AC3 proof — config auto-migration/aliasing for group-shaped configs

## Where the alias lives

`src/server/configMigrations.ts` is the versioned migration engine
(`schemaVersion` chain: 0 → 1 → 2, `CURRENT_SCHEMA_VERSION = 2`,
`configMigrations.ts:20`). The only group-shaped legacy input this app has
ever supported is the schemaVersion-0 preset list, and it now accepts either
spelling:

```
configMigrations.ts:82-109
// The pre-users/accounts config: a single household + groups[] whose memberIds
// are Jellyfin UUIDs. ... A schemaVersion-0 source may spell this preset list
// `parties[]` (post-rename terminology) or `groups[]` (pre-rename); both are
// accepted — see legacyGroupsFrom below.
interface LegacyConfig {
  household?: { username?: string; password?: string };
  groups?: LegacyGroup[];
  // Alias for `groups[]` using the current "party" terminology. A source
  // written by/for a post-rename deployer may use this key instead; if BOTH
  // are present, `parties` wins (it is the more specific/intentional key).
  parties?: LegacyGroup[];
  ...
}

// The legacy preset list, accepting either spelling: `parties[]` (current
// terminology) takes precedence when present, else the pre-rename `groups[]`.
function legacyGroupsFrom(legacy: LegacyConfig): LegacyGroup[] {
  return legacy.parties ?? legacy.groups ?? [];
}
```

`migrate0to1` (the only migration that ever reads the legacy preset list)
calls `legacyGroupsFrom(legacy)` (`configMigrations.ts:122`) rather than
`legacy.groups` directly, so every consumer of the legacy list — user-name
resolution, the drop-warning — sees whichever spelling is present, with
`parties[]` preferred. **No new `schemaVersion` was introduced**: this is a
same-version input alias, not a new migration step, so there is no schema
rollback risk and no manual migration step for operators — the existing v0
config is read exactly as it was authored, in either spelling.

The drop-warning explicitly names both spellings and completes the
terminology shift in its own language:

```
configMigrations.ts:175-177
  if (legacyGroups.length > 0) {
    warnWith(ctx, 'dropped obsolete legacy groups[]/parties[] presets (parties are now formed live in the UI).');
  }
```

## Unit coverage: `src/server/configMigrations.test.ts`

Positive/precedence/warning coverage of the alias, added alongside the
pre-existing `groups[]` tests (which are untouched and still pass, proving
the old spelling keeps working):

- `'migrate0to1 carries over playback/recommendations and drops obsolete
  groups[] (warns)'` (line 95) — pre-existing `groups[]`-only case still
  passes, still warns, still drops the key from the result
  (`assert.equal('groups' in result, false)`, line 108).
- `'migrate0to1 accepts parties[] as an alias for the legacy groups[] preset
  list'` (line 113) — a config spelled with `parties: [...]` (no `groups`
  key at all) resolves to the same `users[]` result as the equivalent
  `groups[]` config.
- `'migrate0to1 prefers parties[] over groups[] when both are present'`
  (line 124) — a config carrying **both** `parties: [{ memberIds:
  ['uuid-a'] }]` and `groups: [{ memberIds: ['uuid-b'] }]` resolves to
  `['Alice']` only (the `parties[]` member), proving `parties[]` wins per
  the documented precedence.
- `'migrate0to1 warns about dropped parties[] presets using party
  terminology'` (line 136) — asserts the warning text matches both
  `/groups\[\]\/parties\[\] presets/` and `/parties are now formed/`
  (line 144), i.e. the warning language itself completed the terminology
  shift, not just the accepted input key.

Two tests exercise the **full** `buildEffectiveConfig` pipeline (0 → 1 → 2)

Both tests assert **identical** resulting shape: `schemaVersion: 2`,

'house' }`, and the same `recommendationCount: 6` — proving the two
spellings are indistinguishable to every downstream consumer of the
migrated config.

## No rollback risk

`runMigrationChain` (`configMigrations.ts:283-320`) only ever throws when
`sourceVersion > CURRENT_SCHEMA_VERSION` (a config declaring a version this
image doesn't understand yet — "run a newer image" case, unrelated to this
alias). Every version at or below `CURRENT_SCHEMA_VERSION` migrates forward
automatically; this alias adds an accepted input spelling at v0 without
touching the version-gating logic at all, so there is no new failure mode
and no rollback path introduced.
